winter-redis-ddc

开箱即用的 Spring Boot Redis 企业级工具包

基于 Redisson 与 Spring Data Redis 构建，提供声明式限流、分布式锁、动态配置中心、布隆过滤器、 Pub/Sub 消息、GEO 操作等企业级 Redis 能力。只需添加依赖并配置连接信息， 即可通过注解或模板 API 快速集成，无需繁琐的样板代码。

GitHub Maven Central

Introduction

项目概览

winter-redis-ddc 是一个面向 Spring Boot 生态的 Redis Starter，旨在将复杂的 Redis 分布式能力 封装为简洁的声明式注解与 fluent 模板 API，让开发者专注于业务逻辑而非基础设施。

设计目标

  • 零配置开箱即用 — 添加依赖即完成集成
  • 声明式编程 — 注解驱动，降低使用门槛
  • 覆盖主流 Redis 场景 — 限流、锁、配置、过滤器、消息
  • 基于 Lua 脚本保证原子性 — 限流与锁操作均为原子执行
  • 运行时热更新 — 动态配置支持不重启修改参数

核心特性一览

  • 三种限流算法：固定窗口 / 滑动窗口 / 令牌桶
  • 五种分布式锁：普通锁 / Fair Lock / 读写锁 / MultiLock / RedLock
  • 动态配置中心：@DCCValue + Redis Pub/Sub 热更新
  • 布隆过滤器：概率性去重，支持本地缓存
  • 模板封装：WinterRedisTemplate + WinterRedissionTemplate
  • Pub/Sub 消息：Topic / Pattern Topic / Reliable Topic
  • 高级数据结构：GEO / HyperLogLog / BitSet / 原子计数器
  • 分布式队列：BlockingQueue / DelayedQueue / PriorityQueue

适用场景

  • 微服务 API 限流防护 — 使用 @RateLimit 注解保护接口
  • 分布式定时任务互斥 — 利用分布式锁防止重复执行
  • 功能开关动态调整 — 通过 @DCCValue 运行时切换降级策略
  • 缓存穿透防护 — 使用布隆过滤器拦截不存在 key
  • 实时消息推送 — 基于 Redis Pub/Sub 实现事件通知
  • 用户签到/统计 — HyperLogLog + BitSet 高效统计

Capabilities

核心能力

starter 将 Spring Boot 自动装配与 Redis 分布式能力深度结合， 提供 8 大类 40+ 个开箱即用的功能点。

01

@RateLimit 声明式限流

通过注解即可为任意 Spring Bean 方法添加限流能力，无需侵入业务代码。 支持 SpEL 动态 Key 表达式。

三种算法：固定窗口（Fixed Window）峰谷控制、滑动窗口（Sliding Window）平滑限制、 令牌桶（Token Bucket）应对突发流量。基于 Lua 脚本在 Redis 服务端原子执行。

Lua 原子执行 AOP 切面 SpEL 动态 Key
02

分布式锁

基于 Redisson 提供完整的分布式锁能力，覆盖多种锁语义场景。

普通可重入锁、Fair Lock（公平锁）、ReadWriteLock（读写分离）、 MultiLock（多实例聚合锁）、RedLock（Redlock 算法）。 提供 executeWithLock() 模板方法自动管理加锁/解锁生命周期。

可重入 Redlock 模板方法 超时控制
03

@DCCValue 动态配置中心

运行时从 Redis 读取配置并注入到 Bean 字段，支持不重启实时更新。

格式 @DCCValue("key:defaultValue")，启动时通过 BeanPostProcessor 扫描所有 Bean，从 Redis 读取并注入。运行时通过 Redis Pub/Sub 发布配置变更消息， 监听器自动更新字段值。支持 AOP 代理对象穿透。

运行时热更新 Pub/Sub 驱动 BeanPostProcessor
04

布隆过滤器

概率性数据结构，高效判断元素是否存在，适合缓存穿透防护场景。

提供 getOrCreateBloomFilter() 自动创建 / 获取过滤器， addToBloomFilter() 添加元素、containsInBloomFilter() 判断存在性。支持指定预期容量与误判率，内置本地缓存加速判断。 提供 deleteBloomFilter() 清理功能。

概率性 本地缓存 O(1) 判断
05

WinterRedisTemplate

封装 Spring Data Redis 的 RedisTemplate，提供 fluent 风格的操作 API。 覆盖 String / Hash / List / Set / ZSet 五大基础数据结构，内置异常处理与日志。

方法级异常捕获、常用操作的便捷封装（如 hPutIfAbsent、 lPushIfPresent 等）、批量操作的原子性保证。 适合基础的 Redis 操作场景。

1820+ 行 异常处理 fluent API
06

WinterRedissionTemplate

封装 RedissonClient，提供 20+ 类高级 Redis 操作，覆盖分布式场景全部需求。

包括分布式锁（5 种）、布隆过滤器、原子计数器（AtomicLong）、 信号量（Semaphore）、CountDownLatch、队列（Blocking / Delayed / Priority）、 Pub/Sub（Topic / Pattern / Reliable）、GEO 地理位置、 HyperLogLog 基数统计、BitSet 位图、批量操作、键空间管理（TTL / expire / persist）。

1850+ 行 20+ 功能 GEO / HLL
07

Pub/Sub 消息

基于 Redisson 的 Topic 机制实现 Redis 发布订阅，支持多种消息模式。

Topic（普通频道）、PatternTopic（通配符订阅）、ReliableTopic（可靠消息， 基于 Redisson ReliableTopic 保证消息不丢失）。 自动序列化/反序列化，泛型消息监听器，支持多消费者负载均衡。

普通频道 通配符 可靠消息 泛型监听

高级数据结构

覆盖 Redis 全部高级数据结构的企业级封装，助力复杂业务场景。

GEO 地理位置：geoAdd / geoDist / geoRadius。 HyperLogLog：pfAdd / pfCount 基数统计。 BitSet：setBit / getBit / bitCount 位运算。 原子计数器：incrementAndGet / decrementAndGet。 信号量：trySetPermits / acquire / release。

GEO HyperLogLog BitSet 原子操作

Patterns

设计模式分布

winter-redis-ddc 在架构中运用了 8 种经典设计模式，分布在各个层次中。

Auto-Configuration config/ + spring.factories Spring Boot Starter 标准模式。通过 spring.factories SPI 注册 AutoConfiguration，@ConditionalOn* 控制 Bean 创建条件，@EnableConfigurationProperties 绑定外部配置。
Template Method / Facade WinterRedisTemplate, WinterRedissionTemplate 封装底层 Redis 客户端（RedisTemplate / RedissonClient）的复杂 API，提供统一且简化的外观接口。方法级异常捕获避免异常传播。
Strategy enums/LimitAlgorithm + lua/*.lua LimitAlgorithm 枚举定义三种限流算法策略，RateLimiterService 根据枚举值选择对应的 Lua 脚本执行。新增算法只需添加枚举值 + Lua 脚本。
AOP (Aspect-Oriented) aop/RateLimitAspect 通过 @Around 切面实现声明式限流的横切关注点。开发者只需 @RateLimit 注解即可获得限流能力，业务代码零侵入。
BeanPostProcessor DynamicConfigCenterBeanPostProcessor Spring 容器扩展点。在每个 Bean 初始化后拦截，扫描 @DCCValue 字段并注入配置值。通过 AopProxyUtils 穿透 AOP 代理。
Observer / Pub-Sub listener/ + Redisson RTopic Redis Pub/Sub 实现配置变更的发布-订阅模式。DynamicConfigCenterAdjustListener 作为观察者接收变更消息，无需轮询。
Lua Scripting RateLimiterService + lua/ 限流逻辑通过 Lua 脚本在 Redis 服务端原子执行，避免并发竞争条件。脚本通过 LuaScriptLoader 缓存，避免重复加载。
Cache-Aside (Memoization) RateLimitAspect (SpEL 缓存), LuaScriptLoader SpEL 解析结果与 Lua 脚本内容均通过 ConcurrentHashMap 缓存，避免重复计算与 I/O，提升运行时性能。

Algorithms

限流算法详解

@RateLimit 支持三种限流算法，分别适用于不同流量特征。所有算法均通过 Lua 脚本在 Redis 服务端 原子执行，无并发安全问题。

固定窗口 LimitAlgorithm.FIXED_WINDOW
fixed_window.lua ~30 行 Redis Key 前缀: ratelimit:fw:
以固定时间窗口（如 1 秒）为粒度进行计数。窗口内请求数未达上限则放行并计数 +1， 已达上限则直接拒绝。窗口切换时（以窗口起始时间戳为 key）自动重置计数器。 特点：实现简单、性能最优，但窗口边界存在流量突刺（两倍峰值）。
INCR 计数 EXPIRE 自动过期 边界突刺 性能最佳 
local key       = KEYS[1]
local limit      = tonumber(ARGV[1])
local windowMs   = tonumber(ARGV[2])
local now        = tonumber(ARGV[3])
local windowKey  = key .. ':' .. math.floor(now / windowMs) .. ':'
local count      = redis.call('INCR', windowKey)
if count == 1 then redis.call('PEXPIRE', windowKey, windowMs) end
if count > limit then return 0 else return 1 end
滑动窗口 LimitAlgorithm.SLIDING_WINDOW
sliding_window.lua ~40 行 Redis Key 前缀: ratelimit:sw:
使用 Redis ZSet（有序集合）记录每个请求的时间戳。每次请求时通过 ZREMRANGEBYSCORE 移除窗口之外的过期记录，ZCARD 统计当前窗口内的请求数量。 特点：彻底消除边界突刺，限流曲线平滑，但内存占用稍高（需存储时间戳）。 适合对流量平滑度要求较高的场景。
ZSet 时间戳记录 ZREMRANGEBYSCORE 清理 ZCARD 计数 零突刺 
local key       = KEYS[1]
local limit      = tonumber(ARGV[1])
local windowMs   = tonumber(ARGV[2])
local now        = tonumber(ARGV[3])
local windowKey  = key .. ':' .. math.floor(now / 1000) .. ':'
local minScore   = now - windowMs
redis.call('ZREMRANGEBYSCORE', windowKey, '-inf', minScore)
local count      = redis.call('ZCARD', windowKey)
if count >= limit then return 0 end
redis.call('ZADD', windowKey, now, now .. ':' .. math.random())
redis.call('PEXPIRE', windowKey, windowMs)
return 1
令牌桶 LimitAlgorithm.TOKEN_BUCKET
token_bucket.lua ~45 行 Redis Key 前缀: ratelimit:tb:
以恒定速率向桶中添加令牌（基于时间差计算），桶满则停止添加。每次请求消耗一个令牌， 令牌不足则拒绝。支持 capacity 参数设置桶容量（突发上限）。 特点：允许一定程度的突发流量（由 capacity 控制），同时保证长期平均速率。 最接近实际业务需求的限流模型，但实现稍复杂。
令牌生成速率控制 桶容量突发 支持突发 长期平均限流 
local key       = KEYS[1]
local rate       = tonumber(ARGV[1])         -- permitsPerSecond
local capacity   = tonumber(ARGV[2])
local now        = tonumber(ARGV[3])
local lastKey    = key .. ':last'
local tokensKey  = key .. ':tokens'
local lastTime   = redis.call('GET', lastKey)
local tokens     = redis.call('GET', tokensKey)
if lastTime == false then lastTime = now end
if tokens == false then tokens = capacity end
tokens = tonumber(tokens)
local elapsed   = math.max(0, now - tonumber(lastTime))
tokens = math.min(capacity, tokens + elapsed * rate / 1000)
if tokens < 1 then return 0 end
tokens = tokens - 1
redis.call('SET', lastKey, now)
redis.call('SET', tokensKey, tokens)
return 1
算法对比 Algorithms Comparison
维度 固定窗口 滑动窗口 令牌桶
流量突刺 窗口边界 2x 无 可控（capacity）
内存占用 低（1 key） 中（ZSet） 低（2 key）
执行性能 最优（1 INCR） 一般（Z* 操作） 优秀（GET/SET）
突发支持 不支持 不支持 支持
实现复杂度 最低 中等 中等
适用场景 简单限流、网关 平滑限流、API 突发流量、业务

声明式限流 — @RateLimit

在任意 Spring Bean 方法上添加注解即可限流 
@RateLimit(
    key = "#userId + ':' + #orderType",
    permitsPerSecond = 5,
    windowSize = 1,
    algorithm = LimitAlgorithm.TOKEN_BUCKET,
    capacity = 10
)
public String createOrder(String userId, String orderType) {
    // 限流保护的业务逻辑
    return orderService.create(userId, orderType);
}

key 支持 SpEL 表达式动态生成；algorithm 可选 FIXED_WINDOW / SLIDING_WINDOW / TOKEN_BUCKET； 限流触发时抛出 RateLimitException（继承 RuntimeException）。

动态配置 — @DCCValue

运行时从 Redis 读取配置，支持自动热更新 
@Component
public class OrderService {

    @DCCValue("order.downgrade:0")
    private String downgradeSwitch;

    @DCCValue("order.timeout:3000")
    private String timeoutMs;

    public void process() {
        // 无需重启，修改 Redis 中 order.downgrade 的值即可生效
        if ("1".equals(downgradeSwitch)) {
            // 降级逻辑
        }
    }
}

格式 @DCCValue("key:defaultValue")，key 对应 Redis 中的 {system}_key。 通过 Redis Pub/Sub 发布 {"attribute":"order.downgrade","value":"1"} 触发热更新， 所有运行实例自动接收变更。

FAQ

常见问题

使用 winter-redis-ddc 过程中可能遇到的典型问题及解答。

如何自定义 RedissonClient 配置？
直接在 application.yml 的 winter-redis-config.redission.* 下配置即可。 若需完全自定义，声明自己的 RedissonClient Bean 并使用 @Primary 注解，Starter 的 @ConditionalOnMissingBean 会自动跳过默认创建。
@RateLimit 支持类级别注解吗？
当前只支持方法级别。若需整个类限流，可在每个方法上添加相同 key 的 @RateLimit， 或自行基于 RateLimiterService 封装类级别拦截器。
@DCCValue 字段更新后，如何触发回调？
当前版本配置更新后仅注入新值，不触发回调方法。若需在值变更时执行逻辑， 建议在业务方法中主动判断字段值，或通过 AOP 监听字段变更。
分布式锁超时后锁自动释放，业务还没执行完怎么办？
Redisson 提供了 Watch Dog 机制：默认 leaseTime 为 -1 时，Redisson 会启动后台线程 定期（leaseTime/3）续期锁的过期时间。使用 executeWithLock() 时建议不传 leaseTime 或传 -1 以启用 Watch Dog 自动续期。
Sliding Window 限流的内存占用如何优化？
滑动窗口使用 ZSet 存储时间戳，每个请求对应一个 ZSet 元素。建议结合业务设置合理的时间窗口 大小（如 1s 而非 1h），窗口越小内存占用越低。也可通过降低请求频率减少 ZSet 基数。
布隆过滤器的误判率如何影响内存？
误判率每降低一个数量级（如 1% → 0.1%），所需内存约增加 50%。公式： bit数 ≈ -insertions * ln(falseProbability) / (ln2)^2。 建议根据业务容忍度合理设置，如缓存穿透防护场景 1% 误判率通常可接受。
Spring Boot 3.x 兼容吗？
Starter 基于 Boot 2.6.11 构建，但坐标不绑定 Boot 版本。 在 Spring Boot 3.x 项目中引用时，需排除传递依赖的 spring-boot-starter-data-redis 并引入与 Boot 3.x 兼容的版本。建议自行验证后再用于生产。
如何贡献代码或报告问题？
项目托管在 GitHub。 欢迎提交 Issue 报告 Bug、Feature Request，或直接提交 Pull Request。 代码风格遵循阿里巴巴 Java 开发手册，提交前请确保通过现有测试。

Caveats

注意事项

限流 Redis Key 命名规则： ratelimit:{算法前缀}:{key}，其中 fw = fixed window， sw = sliding window，tb = token bucket。 多个应用共享 Redis 时注意 key 冲突，建议通过 SpEL 表达式加入业务前缀。
动态配置 Key 格式： {system}_{attributeName}。例如 system=my-service、 attribute=downgradeSwitch 时，Redis 中的 key 为 my-service_downgradeSwitch。首次访问不存在时自动使用 @DCCValue("key:default") 中的默认值并写入 Redis。
配置变更 Topic 命名： DYNAMIC_CONFIG_CENTER_REDIS_TOPIC_{system}。 通过 Redisson RTopic 发布 AttributeVO JSON 消息即可触发全实例热更新。
AOP 代理兼容性： DynamicConfigCenterService 使用 Redisson 的 AopProxyUtils 工具类来穿透 Spring AOP 代理（JDK 动态代理 & CGLIB）， 确保 @DCCValue 字段注入对于被 @Transactional、 @Cacheable 等注解的方法也正常工作。
限流注解 SpEL 缓存： RateLimitAspect 内部会缓存解析后的 SpEL 表达式 （使用 ConcurrentHashMap），避免每次请求重复解析。 缓存 key 为切点方法全限定名。如果遇到动态 SpEL 不生效问题，检查表达式是否正确。
Spring Boot 版本兼容： Starter 基于 2.6.11 构建，但兼容 Spring Boot 3.x（需自行测试）。 对于 Spring Boot 3.x 项目，可能需要调整 Jackson 相关依赖版本。
Lua 脚本加载机制： 三个限流 Lua 脚本（fixed_window.lua / sliding_window.lua / token_bucket.lua）存放在 resources/lua/ 目录下， 通过 LuaScriptLoader 在第一次使用时加载并缓存到 ConcurrentHashMap。 脚本使用 LongCodec.INSTANCE 编解码，保证返回 Long 类型结果。
线程安全： 所有服务类均设计为无状态 Spring Bean，可安全地在多线程环境下使用。 WinterRedissionTemplate 中的 RLock / RSemaphore / RCountDownLatch 等 Redisson 对象本身就具备线程安全性。
分布式锁使用建议： executeWithLock() 方法会自动处理加锁、等待、解锁、异常释放。 建议优先使用模板方法而非手动调用 lock() / unlock()， 避免忘记解锁导致死锁。锁的 watch dog 自动续期由 Redisson 内部处理。
布隆过滤器参数选择： getOrCreateBloomFilter(name, expectedInsertions, falseProbability) 中，expectedInsertions = 预期插入量，falseProbability = 期望误判率（0~1）。 falseProbability 越小占用内存越大。例如 0.01（1% 误判率）与 0.001（0.1%）的内存差异约 50%。
